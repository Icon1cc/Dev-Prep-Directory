## Why Practice Matters

```
Reading about system design ≠ Doing system design

Practice helps you:
├── Think under time pressure
├── Communicate clearly
├── Cover all aspects systematically
├── Get feedback on blind spots
└── Build confidence
```

## Tier 1: Mock Interview Platforms

### Pramp
**URL**: https://www.pramp.com/
**Cost**: FREE
**Format**: Peer-to-peer mock interviews

**How It Works**:
1. Sign up and select "System Design"
2. Get matched with another user
3. Take turns: 30 min interviewer, 30 min interviewee
4. Give each other feedback

**Pros**:
- Free
- Real interview simulation
- Feedback from peers
- Schedule when convenient

**Cons**:
- Partner quality varies
- Limited system design problems

**Best For**: Regular practice, getting comfortable talking

---

### Interviewing.io
**URL**: https://interviewing.io/
**Cost**: Free for practice, paid for expert feedback
**Format**: Anonymous mock interviews

**How It Works**:
1. Practice anonymously with peers (free)
2. Or pay for interviews with engineers from FAANG

**Pros**:
- Anonymous (less stressful)
- Can get expert feedback (paid)
- Recording available

**Cons**:
- Expert sessions are expensive
- Wait times can be long

**Best For**: Realistic practice with anonymity

---

## Mock Interview Tips

### As Interviewee

```
DO:
├── Clarify requirements first (2-3 min)
├── State assumptions explicitly
├── Think out loud constantly
├── Start simple, add complexity
├── Discuss trade-offs
├── Watch the time

DON'T:
├── Jump into solution immediately
├── Stay silent while thinking
├── Over-engineer from start
├── Forget non-functional requirements
├── Run out of time before scaling discussion
```

### As Interviewer (helps you learn too!)

```
DO:
├── Ask clarifying questions
├── Give hints when stuck
├── Push for trade-off discussions
├── Ask "what if" scenarios
├── Give constructive feedback

DON'T:
├── Just say "correct" or "wrong"
├── Let them run out of time
├── Be overly harsh
├── Focus only on what's missing
```

## Getting Better Feedback

### Questions to Ask Your Mock Partner

1. "Did I cover all major components?"
2. "Was my explanation clear?"
3. "Did I talk too much or too little?"
4. "What would you have done differently?"
5. "What was the weakest part of my design?"
6. "Did I handle the trade-offs well?"

### Self-Evaluation Checklist

After each practice, rate yourself:

```
□ Did I clarify requirements? (1-5)
□ Did I do capacity estimation? (1-5)
□ Did I design APIs? (1-5)
□ Did I choose appropriate storage? (1-5)
□ Did I address scalability? (1-5)
□ Did I discuss trade-offs? (1-5)
□ Did I manage time well? (1-5)
□ Did I communicate clearly? (1-5)
```
